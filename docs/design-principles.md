# navi 視覚デザイン原則

> **身体性（physical presence）を背骨にした、navi の見た目を貫く原則集。**
> 個別の UI 仕様書ではなく、**新しい UI を考えるときの判断基準**。
> 「なぜそうするか（原則）」を主、「現状どう適用したか（値）」を従とする。
> 具体の現行値は §後半の「適用リファレンス」と playground に置き、ここでは**転用できる抽象**を述べる。
>
> 用語の正は `docs/glossary.md`、思想の背景は `docs/concept.md`。
> 最初の適用例は**ステータスカード**（`examples/playground`）。以降の UI もこの原則から導く。

---

## 0. 何を作っているのか — 2 つの引力

navi は通知ビューアの皮をかぶっているが、目指すのは**デスクトップに棲む存在（ネットナビ）**。
だから視覚デザインは、相反する 2 つの引力を**同時に**満たす必要がある。

- **身体性**：画面に貼られた絵ではなく、**その場所に実在する物**であること。光・影・重さ・位置の帰結を持つ。
- **静けさ**：おしゃべりなキャラではなく、**静かな情報レイヤー（Live Activity）**であること。割り込まない・読みやすい。

この 2 つの緊張関係が、以下すべての原則の出どころ。迷ったら「実在感を上げるか」「静けさを壊さないか」に還す。

---

## 1. 原則

各原則は **言明 → なぜ → 新しい UI への当て方（テスト） → 現適用** の順。テストは「新要素を作るときに自問する質問」。

### P1. 身体性 — 画面の「どこか」に実在させる

- **言明**：要素は「貼られた絵」ではなく「空間に置かれた物」として扱う。光・影・重さ・位置の帰結を持たせる。
- **なぜ**：実在感は親しみと信頼の土台。影は装飾ではなく**接地（どこに座っているか）の証拠**。物として振る舞うほど「居る」と感じられる。
- **テスト**：
  - この要素は光源に対してどう振る舞う？ 影はどこへ落ちる？
  - 画面内の**位置**が変わったら、見え方も変わるべきでは？（変わらないなら、それは「絵」かもしれない）
  - 重さ（density・elevation）は周囲と整合している？
- **現適用**：接地影（elevation）、positional shadow（位置で影が倒れる）、raking light（端ほど影が寝る）。

### P2. 静かな情報レイヤー — Live Activity であって発話ではない

- **言明**：既定は**状態駆動・連続**（「今こうである」をその場で更新し、続く限り残る）。イベント駆動の割り込み（toast）や発話（chat）に**しない**。
- **なぜ**：常駐物が騒ぐと疲れる。navi の価値は「ちらっと見れば状態が分かる」こと。静けさが情報価値を担保する。
- **テスト**：
  - これは「起きた事を点で知らせる」のか「今の状態を居続けて示す」のか。既定は後者。
  - 人の操作・判断を要する瞬間だけが**通知相当**（＝アテンション）。それ以外で前に出ない。
- **現適用**：ステータスカードは状態をその場更新／尻尾廃止／`waiting_approval`・`error` のみ通知扱い。

### P3. 形が先、色は意味のために取っておく

- **言明**：種類・出自は**形（アイコン・レイアウト）**で伝える。色は**状態・意味の予算**として温存し、装飾に使わない。
- **なぜ**：色を賑やかしに浪費すると、本当に色で語るべき**状態（accent）**が埋もれる。視線が割れる。
- **テスト**：この色は「意味」を運んでいるか、ただ賑やかなだけか。意味でないなら中立トーンへ。
- **現適用**：ソースバッジ既定は **mono（モノクロ統一）**。色は state accent に予約。

### P4. 光学的真実 ＞ 幾何学的真実

- **言明**：数値上の中心や枠寸法ではなく、**人の目が感じる中心・大きさ・重さ**で揃える。
- **なぜ**：「箱は揃っているのに揃って見えない」は頻発する。知覚は幾何学と一致しない（cap 高、塗り面積、占有率）。
- **テスト**：揃えたつもりがズレて見えるとき、line-box でなく cap 高で、枠でなく**視覚的占有率/重量**で測り直したか。
- **現適用**：アイコン⇔テキストは cap-height 中心整列／バッジは viewBox 占有率で glyph スケール補正。

### P5. 一貫した物理 — 値ではなく「関係」を設計する

- **言明**：値を孤立して決めない。**要素間の関係**（影の到達と gap、光源とウィンドウ位置）こそが「世界の法則」。法則を先に決め、値はそこから導く。
- **なぜ**：関係が破れた瞬間に「不自然・汚れ・違和感」が出る。法則で考えれば再現・転用できる（事例の暗記にならない）。
- **テスト**：
  - この値は隣の何と関係する？ それを超えると何が壊れて見える？
  - 1 つの物理（単一光源・一貫した重力）で全要素を説明できるか。例外を増やしていないか。
- **現適用**：**影の到達 ≦ カード間 gap**（超えると帯化＝汚れ → 負 spread で内側に絞る）／全カードが 1 つの光源ベクトルを共有。

## 2. 新しい UI を作るときのチェックリスト

原則を質問に落としたもの。新要素（メニュー、トレイ、設定面、別種のカード…）を起こすときに通す。

1. **身体性 (P1)**：これは空間に置かれた物か？ 光・影・位置の帰結を持たせたか。位置で見え方が変わるべきでは？
2. **静けさ (P2)**：状態表示か割り込みか。前に出る権利（アテンション）があるのは本当にこの瞬間だけか。
3. **色の予算 (P3)**：その色は意味を運ぶか。出自・種類は形で語れているか。色を装飾に使っていないか。
4. **光学 (P4)**：揃えた基準は幾何（枠・line-box）か知覚（cap 高・占有率・視覚重量）か。目で見て揃っているか。
5. **関係 (P5)**：各値は周囲の何と関係するか言えるか。1 つの物理で説明できるか。破ると何が「汚れ」て見えるか。
6. **2 つの引力 (P0)**：この案は実在感を上げるか／静けさを壊さないか。トレードオフを自覚しているか。

> 値そのものより、**どの原則のどのテストで、なぜその値にしたか**を PR・コミットに残す。
> （例：「cap-height 中心整列 (P4) で badge top=6px」「影到達≦gap (P5) で spread=-8px」）

## 3. 原則 → 適用のつながり

本書は**原則が主**。具体値・式・コード識別子の**完全版は `docs/status-card-design.md`**（最初の適用例＝
ステータスカード）に置く。ここでは「どの原則が、どこに、どう効いているか」の橋渡しだけを示す。

| 適用箇所 | 何をしたか（要約） | 効いている原則 |
|---|---|---|
| ソースバッジ既定 = mono | 色は state に譲り、出自は形で伝える | P3 |
| バッジの glyph スケール補正 | 枠でなく viewBox 占有率を揃える（Codex は full-bleed で重い） | P4 |
| バッジ `top` = cap 中心整列 | line-box でなく cap-height の中心で揃える | P4 |
| カードの接地影 | 背景から持ち上げる単一・真上光源の elevation | P1 |
| 影の負 spread | 影の到達 ≦ gap に絞る（越えると帯化＝汚れ） | P5 |
| positional shadow | ウィンドウ位置で影が倒れる＝画面のどこに居るかを示す | P1 |
| raking light | 端ほど影が長く・柔らかく・薄い（斜め光） | P1 |

> 新 UI はこの表の値を真似るのではなく、§1–2 から自分で導く。導いた結果は同じ形式で
> `docs/status-card-design.md` 側へ追記する（原則＝本書／具体＝あちら、の分担を保つ）。
